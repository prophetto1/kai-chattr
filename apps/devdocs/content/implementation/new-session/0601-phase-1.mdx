**Goal:** Build and prove the New Session runtime control plane in legacy `E:/chattr`, then extract only the locked core pieces into `E:/kai-chattr`.

**Architecture:** The parent Chattr instance owns child instance lifecycle only: port allocation, child process launch, health, open-browser, status, and stop. The child Chattr instance owns its own chat room, MCP ports, worker launch, worker identity, and browser terminal runtime.

**Tech stack:** FastAPI, Python 3.11, existing static frontend, MCP bridge, JSONL event streams, OpenTelemetry, FastAPI WebSockets, dependency-gated `xterm.js`, dependency-gated Windows PTY backend.

**Status:** V3 draft for re-evaluation. Not approved for implementation until the pre-implementation review passes.

## Pre-Implementation Contract

No major product, API, observability, persistence, dependency, event-stream, or inventory decision may be improvised during implementation.

If a locked item below is wrong, implementation must stop and this plan must be revised before code changes continue.

## System Shape

```text
Parent browser on :8300
  |
  | calls parent instance API
  v
Parent Chattr process
  - owns child process handle
  - owns child port set
  - owns child health and stop
  - writes instance lifecycle audit events
  |
  | launches child with CHATTR_* overrides
  v
Child Chattr process on :8400 style port set
  - owns child chat room
  - owns child MCP HTTP/SSE ports
  - owns worker launch and registration
  - owns browser terminal sessions
  - emits child runtime events
```

There is no cycle: the parent may request child actions through the child API after health passes, but parent state must not become the source of truth for child worker internals.

## Locked Product Decisions

1. Build this tranche in `E:/chattr` first.
2. Use `E:/kai-chattr` as the clean destination after the tranche is proven.
3. New Session means a new local Chattr instance, not clearing the current chat.
4. Parent owns child instance lifecycle only.
5. Child owns worker lifecycle and browser terminal runtime.
6. P0 source of record is parent in-memory `InstanceRegistry` plus JSONL audit.
7. P0 has no restart recovery.
8. P0 has no SQLite registry and no database migration.
9. P0 uses a separate `chattr.instance_event.v1` stream.
10. Existing `chattr.runtime_event.v1` remains unchanged in P0.
11. The left Workbench owns the New Session entry point.
12. Existing workflow-template `Start a session` is renamed or relocated before New Session is added.
13. Right rail remains the four existing MCP-backed tabs: Rules, Jobs, Locked, Pinned.
14. Runtime MCP tools may exist for agents, but no fifth right-rail tab is added in this plan.
15. No full legacy repo copy into `E:/kai-chattr`.

## Parent Endpoint Contracts

All parent endpoints use:

- Auth: loopback client and existing browser session token.
- Header: `X-Session-Token`.
- Raw command rule: browser never sends raw argv, cwd, env, shell, token, or command text.
- Error shape: `{ "ok": false, "error": { "code": string, "message": string, "step": string } }`.
- Success shape: `{ "ok": true, ... }`.

### `GET /api/instances`

- Request: no body.
- Response: `{ "ok": true, "instances": InstanceState[] }`.
- Errors: `403 invalid_session`, `403 non_loopback`.
- Side effects: none.
- Test: lists empty array before any launch; lists parent-known running, stopped, and failed states after state changes.

### `GET /api/instances/ports/available`

- Request query: optional `base_port`.
- Response: `{ "ok": true, "port_set": { "web": 8400, "mcp_http": 8401, "mcp_sse": 8402 }, "available": true }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `409 ports_unavailable`.
- Side effects: none.
- Test: rejects occupied web, MCP HTTP, or MCP SSE port.

### `POST /api/instances`

- Request: `{ "base_port": 8400, "open_browser": true, "worker_profiles": ["agent.codex"] }`.
- Request defaults: if `base_port` is omitted, server picks the next available block.
- Response: `{ "ok": true, "instance": InstanceState }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `409 ports_unavailable`, `500 spawn_failed`, `502 health_check_failed`.
- Cleanup: failed spawn or failed health must stop the child process if present, release the port reservation, and mark the instance `failed` without a running process leak.
- Test: verifies argv includes only server-built `run.py` flags and isolated paths.

### `GET /api/instances/{instance_id}`

- Request: no body.
- Response: `{ "ok": true, "instance": InstanceState }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `404 instance_not_found`.
- Side effects: may refresh `last_health_at` only if a health probe is explicitly requested by query.
- Test: unknown instance returns `404`; known failed instance includes `error`.

### `POST /api/instances/{instance_id}/open-browser`

- Request: `{ "path": "/" }`.
- Response: `{ "ok": true, "url": "http://127.0.0.1:8400/" }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `404 instance_not_found`, `409 instance_not_running`, `500 browser_open_failed`.
- Side effects: opens browser on the server host only after URL validation.
- Test: rejects external URLs and path values that are not relative child paths.

### `POST /api/instances/{instance_id}/workers`

- Request: `{ "profiles": ["agent.codex"], "attach_terminal": true }`.
- Response: `{ "ok": true, "workers": WorkerState[] }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `404 instance_not_found`, `409 instance_not_running`, `502 child_api_unreachable`, `502 child_worker_start_failed`.
- Ownership: parent proxies the request to the child; child owns worker process state.
- Test: parent does not construct raw worker argv; child launcher allowlist decides.

### `POST /api/instances/{instance_id}/stop`

- Request: `{ "mode": "graceful" }`.
- Response: `{ "ok": true, "instance": InstanceState }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `404 instance_not_found`, `409 stop_already_in_progress`, `500 stop_failed`.
- Cleanup: stops only the child process owned by this parent registry entry.
- Test: cannot stop unknown pid or non-owned process.

## Child Endpoint Contracts

All child HTTP endpoints use:

- Auth: loopback client and existing child browser session token.
- Header: `X-Session-Token`.
- Raw command rule: browser never sends raw argv, cwd, env, shell, token, or command text.
- Error shape: `{ "ok": false, "error": { "code": string, "message": string, "step": string } }`.
- Success shape: `{ "ok": true, ... }`.

The child process is the source of truth for worker and terminal state. The parent may proxy requests but must not construct child worker argv or mutate child worker internals.

### `POST /api/workers`

- Request: `{ "profiles": ["agent.codex"], "attach_terminal": true }`.
- Request rule: every profile id must exist in the child launcher allowlist.
- Response: `{ "ok": true, "workers": WorkerState[] }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `400 invalid_profile`, `409 worker_already_starting`, `500 worker_spawn_failed`, `500 terminal_attach_failed`.
- Side effects: starts child-owned worker processes from launcher profiles; creates child-owned terminal sessions only when `attach_terminal = true`.
- Touched state: child worker registry and child terminal registry.
- Cleanup: failed worker spawn must not leave a registered running worker; failed terminal attach must either stop the new worker or return a worker with `terminal_id = null` and `error.step = "terminal_attach"` according to the implementation's explicit branch.
- Test: rejects raw argv/env/cwd fields; rejects unknown profile; starts an allowlisted worker; returns `WorkerState`; does not weaken `visible_terminal` safety.

### `GET /api/workers/{worker_id}`

- Request: no body.
- Response: `{ "ok": true, "worker": WorkerState }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `404 worker_not_found`.
- Side effects: none.
- Touched state: reads child worker registry only.
- Cleanup: none.
- Test: unknown worker returns `404`; known worker returns pid, status, profile id, and terminal id without raw command or env fields.

### `GET /api/terminal-sessions/{terminal_id}`

- Request: no body.
- Response: `{ "ok": true, "terminal": TerminalState, "ws_path": "/api/browser-terminal/{terminal_id}/ws" }`.
- Errors: `403 invalid_session`, `403 non_loopback`, `404 terminal_not_found`, `409 terminal_not_attachable`.
- Side effects: none.
- Touched state: reads child terminal registry only.
- Cleanup: none.
- Test: unknown terminal returns `404`; known terminal returns metadata only and never returns raw terminal bytes.

### `WebSocket /api/browser-terminal/{terminal_id}/ws`

- Auth protocol: first client message within two seconds must be `{ "type": "auth", "session_token": string }`; tokens must not be logged or emitted.
- Client messages after auth: `{ "type": "input", "data": string }`, `{ "type": "resize", "rows": number, "cols": number }`, `{ "type": "close", "mode": "detach" | "terminate" }`, `{ "type": "ping" }`.
- Server messages: `{ "type": "auth_ok" }`, `{ "type": "output", "data": string }`, `{ "type": "status", "status": string }`, `{ "type": "error", "code": string, "message": string }`, `{ "type": "pong" }`.
- Errors: `403 invalid_session`, `404 terminal_not_found`, `409 terminal_not_attachable`, `500 pty_attach_failed`, `500 pty_io_failed`.
- Side effects: attaches browser IO to a child-owned PTY; resize updates terminal rows and columns; close detaches or terminates according to message mode.
- Touched state: child terminal registry and PTY manager.
- Cleanup: socket disconnect detaches from browser; terminate mode stops the PTY only if the terminal is child-owned and no ownership conflict exists.
- Test: bad auth rejected; unknown terminal rejected; input reaches PTY; PTY output reaches browser; resize updates rows and cols; close path cleans attachment state.

## Persistence Contract

| Area | P0 decision |
|---|---|
| Live instance state | In-memory parent `InstanceRegistry`. |
| Worker state | Child-owned in-memory state. Parent stores only child-returned summary. |
| Terminal state | Child-owned in-memory state. |
| Audit history | Parent writes `data/instance_events.jsonl` using schema `chattr.instance_event.v1`. |

| Restart recovery | Not implemented in P0. Parent restart loses live registry and does not try to reclaim old child processes. |

| SQLite | Not used in P0. |

Runtime-created instance dirs under `data/instances/{instance_id}` require explicit implementation approval under the `E:/chattr` file-creation rule before code execution starts.

| Log | `chattr.instance.stopped` | after stop cleanup | Records parent-owned cleanup result. | Includes `instance_id`, `pid_present`, and cleanup result. | `tests/test_instances_api.py` asserts stop branch. |
| Log | `chattr.worker.requested` | before parent-to-child worker proxy | Records confirmed worker request without raw argv. | Includes `instance_id`, `profiles_count`, and `attach_terminal`. | `tests/test_instances_api.py` asserts proxy request. |
| Log | `chattr.worker.started` | after child worker start | Records child-owned worker creation. | Includes `worker_id`, `profile_id`, `instance_id`, and `status`. | `tests/test_child_workers_api.py` asserts worker launch. |
| Log | `chattr.terminal.closed` | after browser terminal close | Records detach or terminate path. | Includes `terminal_id`, `worker_id`, and close mode. | `tests/test_browser_terminal_api.py` asserts close cleanup. |

Allowed trace and metric attributes: `instance_id`, `profile_id`, `status`, `result`, `port`, `pid_present`, `http.status_code`, `step`.

Forbidden trace and metric attributes: raw command text, tokens, env values, full stdout/stderr, raw terminal bytes, secret names or values, user prompt text, full filesystem paths.

## Explicit Risks And Accepted Limitations

| Risk | Accepted limitation | Mitigation in this plan | Verification |
|---|---|---|---|
| No P0 restart recovery | Parent restart loses live `InstanceRegistry` and does not reclaim old child processes. | P0 records JSONL audit only; restart recovery stays behind DG3. | Completion criteria require no SQLite or durable registry in P0. |
| Orphaned child process on unexpected parent exit | A child process may outlive a crashed parent. | Normal stop path owns cleanup; rollback requires checking active child process before deleting dirs. | `tests/test_instance_manager.py` covers graceful failure cleanup and owned stop. |
| Port allocation race | A port can become occupied after availability preview. | `POST /api/instances` rechecks ports immediately before spawn and returns `409 ports_unavailable`. | `tests/test_instance_manager.py` and `tests/test_instances_api.py` cover occupied ports. |
| Windows browser-open behavior | Browser open can fail or open on the wrong host if URL validation is weak. | Parent opens only validated loopback child URLs from server-built paths. | `tests/test_instances_api.py` covers external URL rejection and open failure. |
| Terminal dependency compatibility | `xterm.js` and the Windows PTY backend may not be approved or compatible. | DG1 and DG2 block real browser terminal implementation until approved. | Dependency gate task records exact dependency decision before code proceeds. |
| Parent-child ownership split | Parent could accidentally become source of truth for child worker internals. | Parent only proxies worker requests; child owns workers and terminals. | Child endpoint contracts and tests assert child-owned state. |
| MCP process-control boundary | MCP tools could become direct process starters instead of proposal tools. | `chat_instance_request` returns proposals only and browser/PM confirms process changes. | `tests/test_runtime_mcp_tools.py` asserts no direct start/stop side effect. |
| Extraction drift from legacy-only code | P0 could become tightly coupled to legacy static UI or launcher internals. | Extraction gate requires a smaller extractable set mapped to backend API, runtime, terminal, MCP, frontend, and observability boundaries. | Final extraction gate checklist must pass before copying into `E:/kai-chattr`. |
| File creation approval drift | Runtime dirs and new source files could be created without PM approval. | Task 0 requires PM approval for new source files and runtime-created `data/instances/*` dirs before code execution. | Task 0 baseline records approval before implementation starts. |
