---
title: "Chat History Stashing"
description: "Lightweight frontend approach for reducing long chat timelines without changing persistence."
---

> Status: saved product/implementation note. This is not an approved implementation plan.

## Problem

Legacy `E:/chattr` chat rooms can become very long because the frontend keeps old chat history mounted in the browser timeline.

The current behavior is:

- websocket history arrives as `history_batch` events
- each message is appended into `#messages`
- each message becomes a live `.message` DOM node
- messages from inactive channels are only hidden with `display: none`
- `history_limit: "all"` maps to a large recent history window per channel

That means the visible scroll, DOM size, and browser work keep growing as a room ages.

## Recommended first cut

Implement DOM stashing, not full virtualization and not backend archival.

Minimal shape:

1. Keep existing persisted history unchanged.
2. Keep websocket history loading unchanged for the first cut.
3. Cache received messages in frontend memory by channel.
4. Render only the latest window per active channel.
5. Replace older rendered messages with one compact stash row.
6. Let the user expand older batches on demand.

Suggested default:

- `300` visible messages per channel
- stash row copy: `Earlier messages stashed - {count} hidden - Show previous 100`
- expansion step: `100` messages

## Why this direction

This is the lowest-risk UX improvement because it avoids the difficult parts:

- no database migration
- no new persistence format
- no message deletion
- no generated summaries
- no virtual-scroll height math
- no change to export/import behavior
- no change to MCP or agent access to persisted history

The feature only changes what the browser keeps mounted.

## Frontend impact

Primary legacy reference files:

- `E:/chattr/static/chat.js`
- `E:/chattr/static/style.css`
- `E:/chattr/static/idt-theme.css`

Expected implementation shape:

- add a small message render-window manager
- route `history_batch` and live `message` events through that manager
- keep `messageRowById` scoped to currently mounted messages
- re-render the active channel when switching channels
- keep inactive-channel messages in memory, not hidden DOM
- preserve `scrollToMessage()` behavior for mounted messages
- for an unmounted target, expand enough prior batches before scrolling

## Non-goals

- Do not implement summaries in the first cut.
- Do not delete or archive messages.
- Do not introduce full virtual scrolling.
- Do not change the backend store.
- Do not change agent-visible history semantics.

## Verification

Minimum checks for an implementation:

- seed a long history with more than the visible window size
- confirm initial DOM has roughly the visible window plus the stash row
- confirm expanding older messages prepends the next batch
- confirm new live messages still append normally
- confirm channel switching does not keep hidden channel DOM mounted
- confirm copy, delete, todo, pin, and job conversion still work on mounted messages
- confirm `scrollToMessage()` handles mounted and newly expanded messages
