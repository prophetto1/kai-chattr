## Right-column dock — tabs & extraction surface

The right side of the chat view is a **tabbed tool dock**. Six tabs: **Planner · Changes · Code (VSCode) · Terminal · Browser · Task List** (Task List shown only when `useTaskList` has tasks).

### Chattr liftout decision — 2026-06-02

- **Bring now:** the right-dock tab shell and **Changes** diff expression.
- **How:** build Chattr-native components from the OpenHands pattern, not raw-copy the OpenHands files.
- **Applied in legacy Chattr:** `E:\chattr\apps\web\src\components\ReferencePrototype.tsx` renders the six-tab dock shell; `E:\chattr\apps\web\src\components\changes\ChangesDock.tsx` adapts the OpenHands diff viewer using Monaco `DiffEditor`, red/green diff colors, file list, `+/-` counts, and old/diff/new view modes.
- **Do not bring now:** raw OpenHands shared primitives, i18n, SaaS/auth/user/org/payment files, or runtime-bound tab files for Code, Browser, Planner, and Task List.
- **Terminal:** keep Chattr's current xterm seam for now; OpenHands `use-terminal.ts` is useful reference but is tied to OpenHands `command-store`.
- **Why:** the browser target needs the OpenHands page/dock expression visible immediately, while runtime seams are rewired later to Chattr backend/WS/MCP.

### The seam (key for extraction)

The dock splits into two layers:

1. **Portable UI shell (~18 files)** — nav + registry + tab wrappers + the 6 tab components + diff-viewer + terminal + browser panel + conversation-store + tab hooks. Copy-able structure; reskin to our tokens.
2. **Per-tab runtime seam (NOT portable — rewire to chattr's backend)** — each tab is a thin UI bound to ONE backend dependency: Terminal → PTY/WS, Code → code-server URL, Browser → screenshot stream, Changes → git API, Planner / Tasks → agent-event → store.

**chattr relevance:** the **Terminal tab + `use-terminal`** is the same seam as the New Session feature (worker terminals in-browser via xterm.js); the **Changes tab** is the CCD diff view wanted in the dock.

### C. Runtime / data seams — REWIRE to chattr's WS + MCP, do NOT copy

Each surface is a thin UI bound to one backend feed. In OpenHands the feeds run over a **socket.io WebSocket** into Zustand stores. The seam = swap that transport + event handlers for chattr's native WS/MCP; the UI above stays.

| Surface | Inbound feed → store | Outbound / HTTP | Query hook |
| --- | --- | --- | --- |
| **Terminal** | WS → `services/observations.ts` (cmd output) + `services/actions.ts` (RUN echo) → `stores/command-store.ts` | `services/terminal-service.ts` builds the `RUN` action | — |
| **Changes / diff** | — | `api/git-service/v1-git-service.api.ts` → `/api/git/changes`, `/api/git/diff` | `hooks/query/use-unified-get-git-changes.ts`, `use-unified-git-diff.ts` |
| **Code (VSCode)** | — | runtime code-server URL | `hooks/query/use-unified-vscode-url.ts` |
| **Browser** | WS → `stores/browser-store.ts` | — | — |
| **Planner** | WS → `stores/conversation-store.ts` (`planContent`) | — | — |
| **Task List** | WS event → store | — | `hooks/use-task-list.ts` |
| **Agent status / "Working"** | WS → `stores/agent-store.ts`, `stores/status-store.ts` | `services/agent-state-service.ts` | — |
| **WS backbone (feeds all stores)** | `contexts/conversation-websocket-context.tsx` (socket.io) → `services/actions.ts` + `services/observations.ts` → the stores | — | — |
| **HTTP base** | — | `api/open-hands-axios.ts` | — |

**Critical nuance — the Terminal is READ-ONLY in OpenHands.** `hooks/use-terminal.ts` creates xterm with `disableStdin: true` and a hidden cursor; it only *renders* `command-store.commands` (agent command output) and never opens a PTY. `services/terminal-service.ts` is a 7-line helper that just builds the outbound `RUN` action — it does **not** stream a shell. So for chattr's **interactive worker terminals** (New Session, xterm.js in-browser) this is a **display reference only**: interactive use needs stdin enabled + a real PTY-over-WS adapter on chattr's side (the New-Session work already flags this).
