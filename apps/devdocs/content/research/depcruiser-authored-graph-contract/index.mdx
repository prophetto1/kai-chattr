## Short answer

Depcruiser migrations are predefined enough to use an authored graph contract.

The runtime stays first-party. This design borrows the graph/node/state shape associated with LangGraph, but it does not adopt LangGraph as an execution runtime, state store, or dependency.

The split is:

```text
skill = human-facing workflow
graph contract = machine-readable lifecycle
hook runtime = enforcement and next-step router
worker = executes the current node only
```

The hook enforces state transitions. The worker does not remember where it is; the hook state tells it which node is active and what output must exist next.

## Why this differs from boundary-triggered deliberation

The companion requirements post, [Boundary-Triggered Deliberation](/research/boundary-triggered-deliberation), is foundationally different.

That system governs arbitrary Claude Code and Codex work from the tool-call boundary. It cannot see the agent's internal reasoning loop, and it did not author the work's phase graph. Therefore it must infer phase structure from observable tool events.

Depcruiser is different. It is a specific skill workflow with known phases, known artifacts, known approval points, and known verification gates. For this case, the graph is not a guess. The graph is the contract.

The general checkpoint system remains inference-based for open-ended work. Depcruiser uses an authored graph because the migration lifecycle is predefined.

## Failure mode

If approval is only prose, the migration skill is not enforceable.

A flat flow like this is too weak:

```text
trace -> manifest -> approval discussion -> port
```

The hook can only prove that JSON exists. It cannot prove that the migration decision was valid, approved, implemented, and verified.

The fix is to split enforcement into lifecycle nodes:

```text
START
  -> discovery
  -> manifest_generation
  -> adjudication
  -> approval
  -> implementation
  -> closure_verification
  -> browser_verification
  -> disarm
END
```

Each node has required inputs, required outputs, and a completion predicate.

## Required nodes

| Node | Purpose | Completion evidence |
| --- | --- | --- |
| `discovery` | Confirm donor repo, target repo, entry files, package managers, and dependency-cruiser availability. | Tooling check output and migration run identity. |
| `manifest_generation` | Generate donor trace and source manifest. | `before-edit.donor.json` and `source-manifest.json`. |
| `adjudication` | Decide what is ported, adapted, cut, or added. | Valid `adjudication.json` with no unresolved decisions. |
| `approval` | Record user approval of the adjudicated migration interpretation. | `approval.json` with approved hashes. |
| `implementation` | Permit guarded target edits only for approved paths. | Target files match approved implementation nodes. |
| `closure_verification` | Prove the target dependency closure is complete. | Depcruiser target trace, `tsc --noEmit`, and strict reconcile pass. |
| `browser_verification` | Prove the migrated page displays correctly. | Playwright evidence and browser verification report. |
| `disarm` | Close the migration only after verification passes. | Active state cleared and audit event appended. |

## Long-lived adjudication

`adjudication` is a phase, not one assistant response.

It can take one response or many. The hook must not advance based on turn count. It must advance only when state says the node is complete.

Example in-progress state:

```json
{
  "currentNode": "adjudication",
  "status": "in_progress",
  "rounds": 4,
  "openQuestions": [
    "Should the donor auth hook become an adapter or a cut?"
  ],
  "proposedDecisions": [],
  "resolvedDecisions": [],
  "readyForApproval": false
}
```

Example complete state:

```json
{
  "currentNode": "adjudication",
  "status": "complete",
  "rounds": 7,
  "readyForApproval": true,
  "outputs": {
    "adjudication": "_migrations/chattr/conversation-tabs/adjudication.json"
  }
}
```

The fixed lifecycle can contain variable-length nodes. Completion is artifact-based, not response-count-based.

## Typed migration records

The manifest needs more than module verdicts. It needs typed records that the hook can enforce.

Example:

```json
{
  "phase": "approved",
  "approvedManifestHash": "sha256:...",
  "nodes": [
    {
      "source": "src/A.tsx",
      "kind": "port",
      "target": "apps/web/src/A.tsx",
      "status": "approved"
    },
    {
      "source": "src/session.ts",
      "kind": "adapter",
      "target": "apps/web/src/session-adapter.ts",
      "status": "approved"
    },
    {
      "source": "src/legacy-analytics.ts",
      "kind": "cut",
      "reason": "not part of target scope",
      "status": "approved"
    }
  ]
}
```

The hook can now enforce actual migration decisions:

```text
No discovery artifacts -> cannot adjudicate.
No adjudication node -> cannot approve.
No approval node or approved hash -> cannot edit target files.
Target edit outside approved paths -> deny.
Cut node appears in target trace -> fail closure verification.
Undeclared target extra appears -> fail unless it is approved as target_extras.
No browser verification -> cannot disarm.
```

## Where the contract lives

The graph contract and enforcement state belong in `E:/hooks`.

```text
E:/hooks/depcruiser-migrations/
  initialize-depcruiser-migrations/SKILL.md
  depcruiser-migrations.mjs
  graph.contract.json
  nodes/
    discovery.json
    manifest-generation.json
    adjudication.json
    approval.json
    implementation.json
    closure-verification.json
    browser-verification.json
    disarm.json
  validators/
  scripts/

E:/hooks/.state/depcruiser-migrations/
  active.json
  runs/<migration-id>/state.json
  runs/<migration-id>/events.jsonl
```

The target repo keeps migration evidence, not runtime control.

```text
<targetRepo>/_migrations/<sourceRepoIdentifier>/<migrationName>/
  before-edit.donor.json
  source-manifest.json
  adjudication.json
  approval.json
  after-edit.target.json
  playwright-report.json
```

So the boundary is:

```text
E:/hooks = control plane, graph contract, validators, active state
target repo = migration evidence and audit artifacts
```

The hook state points at target-repo evidence paths. The target repo does not own the enforcement state.

## Hook behavior

The hook enforces current-node rules deterministically.

```text
If current node outputs are missing:
  keep worker on the current node.

If current node outputs exist and validate:
  advance active state to the next node.

If worker attempts future-node work:
  deny and inject the current required node.

If approval node is missing:
  deny guarded target edits.

If implementation node is active:
  allow only approved target paths.
```

Example injected steer:

```text
Current node: adjudication.
Required output: adjudication.json.
Next allowed action: produce or revise the migration adjudication summary.
Target edits are blocked.
```

## Not the only node implementation

This depcruiser graph is one node-based implementation pattern, not the only possible one.

Use authored graph contracts when a skill has a predefined lifecycle:

```text
depcruiser-migrations = authored graph contract
other predefined hook bundles = their own graph contracts and state dirs
```

Use inferred phase/window contracts when the work is open-ended:

```text
boundary-triggered-deliberation = inferred phase/window contract
arbitrary Claude/Codex task = observed from tool-boundary state
```

That keeps the architecture clean. Authored graphs are allowed where the workflow is predefined. Inferred graphs remain the correct model for open-ended agent work.

## External reference role

LangGraph remains a reference for the shape of the state contract: nodes, edges, durable state, and human-in-the-loop pause/resume.

It is not the runtime for this system.
