# Migration · Terminal runtime

{/* Hand-authored design — NOT generated from contracts. Status: drafting. */}

The runtime that ends the separate-OS-window problem. **CLI agents stay the
runtime** — we keep chattr's external-CLI execution model and only upgrade how
the terminal is *presented*, using OpenHands' `xterm.js`.

## The pipeline

```text
Agent CLI            PTY (ConPTY / pywinpty)        Transport            UI
claude_wrap · codex  ──drives──▶  no GUI window  ──raw bytes + ANSI──▶  xterm.js
keystrokes & output            real pseudo-terminal    over WebSocket     (RightDock tab)
```

- A real **pseudo-terminal** owns the agent process — **ConPTY** (via
  `pywinpty`) on Windows, `pty` on POSIX. **No OS console window is ever
  spawned.**
- Raw bytes + ANSI stream over a **WebSocket** to an embedded **`xterm.js`**
  panel (the same component VS Code, Cursor, and cloud IDEs use) — borrowed MIT
  from the OpenHands frontend.
- This replaces today's brittle mechanism (a real OS console + Win32 keystroke
  injection + screen-buffer scraping).

## Approvals are elevated, not buried

On a gated command the PTY **pauses** and emits a structured
`terminal.attention_needed` event (already in the `chattr.runtime_event.v1`
vocabulary). The UI overlays a **native Approve / Deny card** — the user's click
is written back to the PTY as the raw `y`/`n` byte. The approval is a
recognisable agent-native moment (per the `frontend` contract's
`legible-message-types` rule), never a `y/N` lost in scrolling text.

## Current → proposed

| Brittle (now) | PTY (proposed) |
|---|---|
| Multiple OS windows | Invisible · rendered in-app |
| Keystroke inject + scrape | Bidirectional byte stream |
| Approvals buried in windows | Elevated UI cards |

## Transport-agnostic

Today the bytes ride a **WebSocket**. A future local **Tauri** build swaps the
transport for native IPC — the `xterm.js` component and the
`chattr.runtime_event.v1` event vocabulary stay **identical**. The desktop build
is therefore a pure transport enhancement, not a rebuild.

## UX direction (concept — not yet locked)

The leaning presentation is **Focus + Ambient**: one conversation rendered like
a document (built for following an agent's reasoning, not skimming a cramped
feed); concurrent sessions breathe in a side rail with one-line status pulses
(running · awaiting-you · ready · idle); the **terminal unfolds inline beneath
the turn that launched it** — execution and its approval live where the work
happened. Concept directions are still *react-and-pick* and not contract-locked.

## Smallest proof slice

The **Terminal tab** in `apps/web`: `xterm.js` mounted as one RightDock tab, fed
by a ConPTY-over-WebSocket bridge, with one `attention_needed` → Approve/Deny
round-trip. Validates the whole tabbed-dock architecture with one real tool.
